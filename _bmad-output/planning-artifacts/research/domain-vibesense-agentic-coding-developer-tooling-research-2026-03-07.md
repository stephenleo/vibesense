"Vibe coding" emerged as a mainstream term in 2025 — programming by intent with AI agents rather than syntax. This cultural shift creates an audience receptive to unconventional developer tools that feel more like games than traditional IDEs.

## Regulatory Requirements

### Applicable Regulations

VibeSense faces regulatory obligations across four domains: platform publishing policies, hardware access standards, data privacy law, and accessibility compliance. None represent blocking concerns, but several require proactive design decisions.

**1. VSCode Marketplace Publishing Requirements (Microsoft)**

The Visual Studio Marketplace enforces the following compliance gates for all extensions:

- **Malware scanning**: Every extension submission is scanned with multiple antivirus engines before publishing. Updates are re-scanned.
- **Dynamic detection**: Extensions are run in a sandboxed clean-room VM to verify runtime behavior before publishing.
- **Publisher trust flow**: Since VS Code v1.97, first-time installs from non-verified publishers trigger a trust confirmation dialog. Publisher verification (blue checkmark) requires domain ownership proof and 6+ months of good standing on the Marketplace.
- **Asset policies**: Extension icons must not be SVGs; README images must use HTTPS URLs.
- **Enterprise policy**: From VS Code v1.106, organizations can block extensions via `extensions.allowed` policy or a `/etc/vscode/policy.json` file on Linux — enterprise customers may restrict VibeSense if it isn't on their allow-list.
- **No formal permission sandboxing**: VS Code does not currently enforce a capability permission system for extensions (unlike Android/iOS). The extension host runs with the same OS-level permissions as VS Code itself. Hardware access (HID, USB) is permitted by default once Electron v25+ enabled Web-* device access.

_Regulatory Body: Microsoft / Visual Studio Marketplace_
_Source: [VSCode Extension Runtime Security](https://code.visualstudio.com/docs/configure/extensions/extension-runtime-security), [Publishing Extensions](https://code.visualstudio.com/api/working-with-extensions/publishing-extension), [Private Marketplace blog](https://code.visualstudio.com/blogs/2025/11/18/privatemarketplace)_

**2. W3C Gamepad API & WebHID API Standards**

- **Gamepad API** (W3C standard, Candidate Recommendation): The primary API for reading controller button/axis state. Requires a **secure context** (HTTPS or localhost). Works in VSCode's WebView (Chromium-based, secure context). No user permission prompt required — passive polling only.
- **WebHID API** (WICG incubator draft — NOT a W3C standard): Required for haptic output, LED control, and DualSense-specific features. **Requires explicit user permission per-origin** via a browser-style permission prompt. The `Permissions-Policy: hid` directive can be used by enterprise admins to block WebHID entirely (relevant for enterprise customers).
- **node-hid** (Node.js, MIT license): For the extension host (non-WebView) side, `node-hid` provides HID device access through Node.js. Electron v25+ makes this available without native binaries in some configurations.

_Regulatory Body: W3C / WICG_
_Source: [W3C Gamepad API](https://www.w3.org/TR/gamepad/), [WebHID API WICG](https://wicg.github.io/webhid/), [node-hid GitHub](https://github.com/node-hid/node-hid), [MDN Permissions-Policy: hid](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Permissions-Policy/hid)_

**DualSense Haptic/LED on PC — Critical Regulatory Detail**
Sony does **not** provide an official PC SDK for DualSense haptic feedback, adaptive triggers, LED bar, or speaker. These features are only officially accessible within PlayStation OS. On PC, they are accessed via the HID protocol using community-maintained libraries:
- [DualSense-Windows](https://github.com/Ohjurot/DualSense-Windows) (C++, MIT license)

### Data Protection and Privacy

Voice input is the most significant regulatory exposure for VibeSense. The regulatory landscape is demanding but navigable with on-device processing.

**GDPR (EU) — Applies to EU users**
- Voice recordings are **personal data (PII)** under GDPR
- Voiceprints qualify as **biometric data** (special category under Article 9) — requiring explicit opt-in consent
- **Data Protection by Design** (Article 25) is a legal requirement, not a best practice
- **Penalties**: Up to €20 million or 4% of annual global turnover for serious violations
- **Required**: Explicit consent before any voice recording, documented data retention windows, right to erasure

**CCPA/CPRA (California) — Applies to California users**
- Voiceprints are classified as biometric data under CCPA
- Requires: clear notice before collection, simple opt-out mechanism, 45-day deletion fulfillment
- **Penalties**: Up to $7,500 per intentional violation

**Compliance Strategy for VibeSense:**
The most robust compliance path is **on-device processing only**:
- Using local Whisper model (open-source, runs locally) means voice data never leaves the device
- No cloud transmission = no data processor agreements, no cross-border transfer rules, dramatically reduced GDPR/CCPA scope
- Browser Speech API (alternative) uses OS-level voice recognition (Windows/macOS) — subject to each OS vendor's privacy policy
- Cloud transcription (OpenAI Whisper API) would require full GDPR compliance stack (DPA, consent flows, retention policies)

**Recommendation**: Default to local Whisper; treat cloud transcription as an opt-in power feature with explicit consent flow.

_Source: [Picovoice — GDPR/CCPA Voice Recognition](https://picovoice.ai/blog/gdpr-ccpa-voice-recognition-privacy/), [Deepgram — Speech-to-Text Privacy](https://deepgram.com/learn/speech-to-text-privacy), [Sensory — Privacy by Design](https://sensory.com/compliance-privacy-by-design/)_

**The Interaction Model Nobody Has Built Yet**

Every player in the ecosystem has solved part of the problem:
- VSCode has the agent session management (screen-only)
- Claude Code has voice (keyboard PTT)
- Copilot has voice (keyboard PTT)
- Controller remappers have button mapping (gaming context, no agent awareness)
- Voice tools have transcription (no hardware integration)

VibeSense combines them all into a single physical experience: pick up a controller, speak a prompt by pressing a button, feel the agent's state through haptics, review output with analog stick scrollback, approve with a button press. No keyboard required. No screen-staring required during agent execution. No context-switching to check if the agent is done.

This is not a feature addition to an existing tool. It is a new interaction paradigm for a new developer workflow that arrived in 2025 and has no native physical interface.
